Vaults & storage

Backups & recovery

Server-side backups are off. Set BACKUP_DIR (and optionally BACKUP_INTERVAL, BACKUP_KEEP) to archive every vault's encrypted blobs and media on a schedule. A backup holds only ciphertext — no keys, no plaintext.
ArchiveCreatedSize

Restore from ?

This replaces every vault on the relay with the contents of this archive — all current encrypted records, media, reminders and devices are overwritten in one step. Sessions are cleared, so every device re-authenticates on next sync.

It does not reach into anyone's devices. Use this for disaster recovery onto a fresh or broken relay; the journald restore CLI is the recommended path when the server can be stopped first.

Delete vault ?

This permanently wipes every encrypted record, media object, reminder, device and session this vault has on the relay. There is no undo and no backup.

It does not reach into anyone's devices — local copies stay where they are, and the same recovery phrase can register again as an empty vault.

Aggregates and storage metadata only. The relay holds opaque ciphertext — it cannot read content, distinguish journals, entries or templates, tell media kinds apart, or attribute the daily counters above to any vault. Vault ids are truncated pseudonymous key hashes.
